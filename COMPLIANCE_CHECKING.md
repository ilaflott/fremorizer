# IOOS Compliance Checker Integration

This document describes the IOOS Compliance Checker integration in fremorizer's CI/CD pipeline.

## Overview

The `compliance_check.yml` workflow automatically validates NetCDF output files generated by fremorizer tests against CF (Climate and Forecast) conventions using the [IOOS Compliance Checker](https://github.com/ioos/compliance-checker).

The workflow uses a **centralized test output gathering approach** that is designed to be reusable by other QA pipelines (such as ESGF-QA). This makes it easy to run multiple validation tools on the same set of test outputs.

## What is the IOOS Compliance Checker?

The IOOS Compliance Checker is a Python-based tool for data providers to check for completeness and community standard compliance of netCDF files. It supports multiple standards including:

- **CF (Climate and Forecast) Conventions** - versions 1.6 through 1.11
- **ACDD (Attribute Convention for Data Discovery)** - versions 1.1 and 1.3
- **IOOS Metadata Profile** - versions 1.1 and 1.2

For fremorizer, we primarily use the **CF conventions checker** as CMORized output must be CF-compliant.

## Workflow Details

### When It Runs

The compliance checking workflow runs:
- On pull requests to any branch
- On manual trigger via `workflow_dispatch`

### What It Does

1. **Sets up environment**: Creates a conda environment with fremorizer dependencies plus compliance-checker
2. **Runs comprehensive tests**: Executes multiple test suites:
   - `test_cmor_run_subtool.py` - Basic CMIP6 tests
   - `test_cmor_run_subtool_further_examples.py` - Extended tests including CMIP7 examples
3. **Gathers outputs centrally**: Uses `.github/scripts/gather_test_outputs.sh` to:
   - Find all NetCDF files from test runs
   - Categorize them by CMIP version (CMIP6, CMIP7, other)
   - Copy to centralized directory with unique names
   - Generate manifest and summary files
4. **Runs compliance checks**: For each category of files:
   - Runs `compliance-checker --test=cf` with normal criteria
   - Generates both text and HTML reports
   - Displays summary results in the workflow log
5. **Uploads artifacts**: Archives both test outputs and compliance reports (retained for 30 days)

### Output Artifacts

The workflow produces two artifacts:

1. **qa-test-outputs**: The centralized test output directory
   - `cmip6/` - CMIP6 test outputs
   - `cmip7/` - CMIP7 test outputs
   - `other/` - Unclassified outputs
   - `manifest.txt` - Mapping of original files to gathered files
   - `SUMMARY.md` - Statistics and usage guide

2. **compliance-checker-reports**: Compliance validation results
   - `cmip6_*_report.html` - Individual CMIP6 file reports
   - `cmip7_*_report.html` - Individual CMIP7 file reports
   - `SUMMARY.md` - Overview of checks performed
   - `TEST_OUTPUTS_SUMMARY.md` - Copy of test outputs summary
   - `manifest.txt` - Copy of file mapping

## Reusable Approach for Other QA Pipelines

The test output gathering script (`.github/scripts/gather_test_outputs.sh`) is designed to be reusable by other QA validation pipelines. This approach provides:

### Benefits

- **Single Source of Truth**: All test outputs are gathered in one place
- **Consistent Organization**: Files categorized by CMIP version
- **Easy Integration**: Other tools can use the same organized outputs
- **Traceability**: Manifest file maps gathered files back to source locations
- **Parallel Validation**: Multiple QA tools can run on the same outputs without re-running tests

### Using the Script

The script can be called from any workflow or local environment:

```bash
# Basic usage - outputs to tmp/qa_test_outputs
bash .github/scripts/gather_test_outputs.sh

# Custom output directory
bash .github/scripts/gather_test_outputs.sh /path/to/output/dir
```

### Script Output Structure

```
tmp/qa_test_outputs/
├── cmip6/              # CMIP6 files (auto-detected)
│   ├── <hash>_file1.nc
│   └── <hash>_file2.nc
├── cmip7/              # CMIP7 files (auto-detected)
│   ├── <hash>_file3.nc
│   └── <hash>_file4.nc
├── other/              # Unclassified files
│   └── <hash>_file5.nc
├── manifest.txt        # Source → gathered file mapping
└── SUMMARY.md          # Statistics and usage examples
```

### Integration with Other QA Tools

#### Example: ESGF-QA

Create a workflow that uses the same gathered outputs:

```yaml
- name: Run tests to generate outputs
  run: pytest fremorizer/tests/ -v
  continue-on-error: true

- name: Gather test outputs
  run: bash .github/scripts/gather_test_outputs.sh tmp/qa_test_outputs

- name: Run ESGF-QA checks
  run: |
    # Check CMIP6 files
    for ncfile in tmp/qa_test_outputs/cmip6/*.nc; do
      esgf-qa --project CMIP6 "$ncfile"
    done

    # Check CMIP7 files
    for ncfile in tmp/qa_test_outputs/cmip7/*.nc; do
      esgf-qa --project CMIP7 "$ncfile"
    done
```

#### Example: Custom Validator

```bash
# Use the gathered outputs with any validation tool
bash .github/scripts/gather_test_outputs.sh tmp/qa_test_outputs

# Validate all CMIP7 files
find tmp/qa_test_outputs/cmip7 -name "*.nc" | while read ncfile; do
  your-custom-validator "$ncfile"
done

# Or validate all files regardless of category
find tmp/qa_test_outputs -name "*.nc" -type f | while read ncfile; do
  your-custom-validator "$ncfile"
done
```

### Key Features

1. **Auto-categorization**: Files are automatically categorized as CMIP6, CMIP7, or other based on:
   - File path patterns (e.g., contains "CMIP6" or "CMIP7")
   - NetCDF global attributes (reads `mip_era` attribute)

2. **Collision Prevention**: Uses MD5 hash prefixes to ensure unique filenames when gathering from multiple directories

3. **Detailed Manifest**: The `manifest.txt` file maps each gathered file back to its original location

4. **Usage Documentation**: The generated `SUMMARY.md` includes examples for common QA tools

## Using Compliance Checker Locally

You can run the compliance checker on your own NetCDF files:

### Installation

```bash
# Via conda (recommended)
conda install -c conda-forge compliance-checker

# Via pip
pip install compliance-checker
```

### Basic Usage

```bash
# Check a single file against CF conventions
compliance-checker --test=cf your_file.nc

# Generate HTML report
compliance-checker --test=cf --format=html --output=report.html your_file.nc

# Use strict criteria
compliance-checker --test=cf --criteria=strict your_file.nc

# Check against specific CF version
compliance-checker --test=cf:1.6 your_file.nc
```

### Understanding Results

Compliance checker categorizes issues into three severity levels:

- **High priority**: Critical issues that violate the standard
- **Medium priority**: Important issues that should be addressed
- **Low priority**: Recommended improvements

Each check shows:
- A score (e.g., "15/20 passed")
- Specific issues found
- References to relevant sections of the CF conventions

## Common CF Convention Issues

When CMORizing data, watch for these common issues:

1. **Missing or incorrect `Conventions` attribute**
   - Should be set to the CF version (e.g., "CF-1.6")

2. **Coordinate variable problems**
   - Missing `standard_name`, `long_name`, or `units`
   - Incorrect axis attributes (`axis="T"`, `axis="X"`, etc.)

3. **Fill value mismatches**
   - `_FillValue` and `missing_value` should be equal
   - Both should match the variable's data type

4. **Time coordinate issues**
   - Missing or invalid `calendar` attribute
   - Incorrect `units` format (should be "days since YYYY-MM-DD")

5. **Missing recommended global attributes**
   - `title`, `institution`, `source`, `history`, `references`

## Interpreting Workflow Results

### Success Case

When the workflow completes successfully, it means:
- Tests ran and generated output files
- Compliance checker successfully analyzed the files
- Reports were generated and uploaded

Note: The workflow will complete even if files have compliance issues. Review the reports to see the actual compliance status.

### Failure Cases

The workflow may fail if:
- Tests fail to generate output files (will show a warning)
- Compliance checker cannot be installed
- File I/O errors occur

## Integration with CMOR

The compliance checker complements CMOR's built-in checks:

- **CMOR**: Enforces CMIP-specific requirements (variable names, dimensions, required attributes)
- **Compliance Checker**: Validates CF convention compliance (coordinate systems, metadata completeness)

Together they ensure output files are both CMIP-compliant and CF-compliant.

## Troubleshooting

### "No NetCDF files found"

This warning appears when tests don't generate output files. Possible causes:
- Test failures prevented output generation
- Tests were skipped
- Output directory path is incorrect

Check the test execution logs earlier in the workflow.

### Compliance checker errors

If compliance-checker itself fails, check:
- The file is a valid NetCDF file
- File permissions allow reading
- The file isn't corrupted

### Understanding scores

A score of "X/Y passed" means:
- X checks passed
- Y total checks were run
- The remaining (Y-X) checks had issues

A perfect score would be "Y/Y passed", but minor issues are common and may not prevent data publication.
